### 1. Про архитектуру
Я выбрал State Machine + DI через VContainer, потому что:

- это даёт строгую изоляцию игровых этапов
- упрощает расширение (можно добавлять новые состояния без изменения старых)
- снижает связность между системами

Альтернативой было бы:
- MonoBehaviour-менеджер с кучей if/flags
но это быстро превращается в "god object" и плохо масштабируется.

### 2. Про State Machine
Я использовал IState + IStateController вместо простого enum-switch, потому что:

- состояние само отвечает за свою логику
- контроллер управляет только переходами
- можно легко тестировать состояния отдельно

### 3. Про AI
- Краткий экскурс по  VContainer
- Логику переходов между состояниями (StateController)
- Структуру GameState enum и его привязку к состояниям
- Базовую архитектуру слоёв (Bootstrap → Controller → States)

AI использовался только для проверки архитектурных идей и поиска потенциальных проблем в реализации.

### 4. Про понимание
Я полностью понимаю:

- как работает DI контейнер VContainer
- как создаются и резолвятся состояния
- как происходит переход между состояниями через IStateController
- как UniTask управляет асинхронностью и cancellation token

### 5. Про "магию"
Частично требует дополнительного углубления:

- внутренняя реализация VContainer (resolution pipeline)
- тонкости cancellation token в цепочках UniTask при переходах состояний

### 6. Про "завтра"
1. Почему State Machine?

Потому что это даёт контролируемый жизненный цикл игры и убирает хаос из MonoBehaviour логики.

2. Почему DI?

Чтобы убрать жёсткие зависимости между системами и упростить расширение проекта.

3. Почему UniTask?

Потому что он проще и легче, чем стандартные корутины, и лучше сочетается с async/await архитектурой.
